# workflow-design-pattern

设计阶段型 AI 工作流 skill 的通用方法论。

---

## 核心洞察

复杂任务的 AI 交互，最怕两个极端：
- **全自动化**：生成即定稿，用户想改时已经晚了（成本高）
- **全手动**：每步都等确认，打断流畅感（体验差）

正确解法：**把「确认节点」放在成本最低的地方**。

---

## 四层架构

```
┌─────────────────────────────────────────┐
│  输入层      — 用户给需求/材料            │
└────────────────┬──────────────────────┘
                  ▼
┌─────────────────────────────────────────┐
│  草案层      — 生成大纲/结构，先展示      │  ← 确认节点（成本最低）
└────────────────┬──────────────────────┘
                  ▼
┌─────────────────────────────────────────┐
│  渲染层      — 执行生成，输出成品        │
└────────────────┬──────────────────────┘
                  ▼
┌─────────────────────────────────────────┐
│  编辑层      — 模块化修改，不全量重写    │
└────────────────┬──────────────────────┘
                  ▼
┌─────────────────────────────────────────┐
│  学习层      — 自动沉淀模板，供后续复用  │  ← 静默执行，不打断用户
└─────────────────────────────────────────┘
```

---

## 每层的关键设计点

### 1. 输入层
- 用户输入可能是模糊的，先补全再继续
- 输出内部 Brief（不对用户展示），作为后续所有步骤的依据

### 2. 草案层（最关键）
- **草案必须是低成本的**：纯文本，不是成品，改动代价极低
- **展示格式要结构化**：表格/大纲树，让用户一眼看清全貌
- **用户指令要具体**：加/减/合并/拆分/确认，指令清晰可执行
- **循环直到确认**：用户可以多轮调整，不用担心重做代价

### 3. 渲染层
- 渲染前先匹配「规则」，不是从零开始
- 规则来自两层：用户规则（优先级高）+ skill 默认规则
- 所有输出引用 tokens，不硬编码

### 4. 编辑层
- 编辑按「意图分类」，每个意图对应一个 reference script
- 执行原则：**只改目标节点，不动其他页面**
- 执行前先展示操作预览，确认后再执行

### 5. 学习层
- 判断「效果好」的标准要明确（老师满意 / 无修改确认）
- 模板记录格式：主题标签 + 结构 + 规则备注
- 匹配时：完全匹配直接用，部分匹配建议确认，无匹配从零生成
- **静默执行**，不打断用户当前工作流

## 规则引擎设计

规则不是「一次性写死」，而是**可累积**的：

```
规则库结构：
├── applies_to        ← 适用范围（通用 / 特定页面类型 / 特定主题）
├── condition         ← 触发条件（可选）
├── rule              ← 规则内容
└── exception         ← 例外条件（如有）
```

**例外条件**是关键设计点：
- "所有字号不小于16px，范文讲解页除外"
- → 规则设为通用，exception 标注「范文讲解页 font-size-min 可降至14px」

## 模块化编辑设计

每个编辑意图对应一个独立模块：

| 模块 | 职责 | 边界 |
|------|------|------|
| 结构模块 | 增/删/拆/合/调序 | 只操作 slide 节点 |
| 样式模块 | 字号/字体/颜色/行距 | 只动 CSS tokens |
| 交互模块 | 加/换/删交互动效 | 只注入/修改 JS + 对应 CSS |
| 图片模块 | 换图/生成图/热区标注 | 只替换图片引用 |
| 风格模块 | 整体换风格 preset | 只换 CSS preset，不动内容 |

## 什么时候用这个 pattern

**适用场景：**
- 用户需求容易变化（老师/客户来回调整）
- 生成结果需要多次迭代才能满意
- 需要记住用户的个人偏好
- 内容有明确的结构（章节/页面/组件）

**不适用场景：**
- 任务是一次性的，不会重复
- 用户只需要结果，不需要编辑能力
- 流程极短，一两步就能完成

## Reference Scripts 推荐拆分方式

按职责分离，不要写成巨无霸：

```
skill/
├── SKILL.md                      ← 主入口（工作流 + 指令路由）
└── references/
    ├── draft-guide.md             ← 草案生成规则
    ├── [intent]-module.md          ← 每个编辑意图一个文件
    ├── rule-book.md               ← 规则库（可用户编辑）
    ├── template-library.md         ← 模板库（自动追加）
    └── generator-rules.md          ← 渲染规范（技术细节）
```

## 验证方法

设计完 skill 后，问自己三个问题：

1. **草案层够低成本吗？** — 改草案的代价应该远低于重生成
2. **编辑层够精准吗？** — 改一个地方会不会意外影响到其他
3. **学习层够安静吗？** — 模板追加不应该打断用户当前操作
